- ✅ **project_secrets token-lookup** (regel 28-41), **PII-filter** (regel 44-86, uitgebreide blacklist + whitelist), **NL timezone hardcoded** (regel 1-2), **retry-wrapper** (regel 200-215), **getMissedDays-fallback** (regel 688-716).
- ❌ **Silent token-failure is de echte root cause van 19 "nooit-gesyncte" projecten**: `validateToken()` (regel 240-249) faalt stil bij elke SOAP Fault. In `performSync` (regel 502-515) wordt bij `tokenValid === false` én `jobId === null` **geen** `sync_logs` of `error_logs` insert gedaan. Dus 19 projecten = 19 stille token-failures tijdens nachtsyncs. ANBO (734) valt hieronder.
- ⚠️ **gesprekstijd_sec=0 voor outbound**: regel 425 extract `parseInt(cleanRecord.bc_gesprekstijd || 0)`. Het veld wordt dus wel geparsed — waarschijnlijk levert BasiCall simpelweg `bc_gesprekstijd=0` voor outbound records omdat daar geen per-call-meting is. Inbound/service krijgt wel waarden. Acceptabel mits ureninzet uit `getIngelogdeTijden` komt, wat al gebeurt.
- ⚠️ **Batches worden niet auto-aangemaakt**: sync leest bestaande batches, maar maakt ze niet aan. Admin moet per nieuw project batches koppelen via `BatchManager.tsx`.

### Actionable fix voor silent token-failure

Patch voor `validateToken` / `performSync` in sync-current.js — voeg een `sync_logs.insert` toe in de failure-branch zodat ANBO en consorten zichtbaar worden:

```js
// performSync regel 504-515 vervangen door:
if (!tokenValid) {
    const msg = `Token ongeldig voor project ${project.name}`;
    await supabase.from('sync_logs').insert({
        project_id: project.id,
        status: 'failed',
        records_synced: 0,
        sync_from: start.toISOString(),
        sync_to: end.toISOString(),
        error_message: msg,
    });
    if (jobId) { ...bestaande code... }
    return;
}
```

Na deze patch: 19 projecten krijgen elke nacht een `failed` sync_log, `mapping_issues`-view of een nieuwe `sync_health`-view kan ze oppervlakken naar admin.

### Voor ANBO (734) specifiek vandaag

Het project heeft token (`project_secrets`), dus óf de token is afgekeurd door BasiCall, óf `Record.getAfgehandeldIds` voor vandaag returnt een SOAP Fault (`validateToken` kan geen onderscheid maken tussen "token fout" en "API-reply fout"). Handmatige test via Admin → SyncManager → "Sync starten" (zet `sync_jobs.status=pending`) is de snelste manier om zichtbaar te maken wat er gebeurt, want bij jobId≠null wordt wél naar `sync_jobs.log_message` geschreven.

### Raw vs processed records
Huidig: **één tabel** `call_records` met `raw_data` JSONB + `beldatum`/`beltijd`/`resultaat` in eigen kolommen. Mapping is runtime (hook `useCallRecords` past `mapping_config` toe). ### PII-scrubbing
Geen migration die NULLs/redact. Briefing noemt 24.438 gescrubde records — matcht niet met huidige 5.461. Waarschijnlijk is er manueel een `DELETE` gedaan of is het getal 24.438 van voor-scrub (en 5.461 is post-scrub). Onkenbaar zonder extra info.

### Timezone
Geen `AT TIME ZONE 'Europe/Amsterdam'` in migrations. VPS schrijft `beldatum` als DD-MM-YYYY-string, trigger converteert naar DATE. Dagsnijding gebeurt dus in whatever timezone de VPS script gebruikt bij het ophalen van `getAfgehandeldIds`. Kas: checken dat VPS timezone op `Europe/Amsterdam` staat (`timedatectl`).

### Audit-log sync-runs
